# AIOps Sentinel - APM Monitoring, LLM Analysis, and AI Output Evaluation

> An experimental project that detects abnormal APM metrics, asks an LLM to analyze the likely cause, and then evaluates the quality of that AI-generated analysis.
> The core QA question is: "Who verifies the result produced by AI?"

## Data Disclaimer

All APM data, host names, WAS names, agent names, thresholds, alerts, and examples in this repository are 100% virtual sample data.

- Data is generated by `apm/mock_generator.py`.
- Identifiers such as `was_sample_01` and `agent_sample_a` are virtual names used for documentation and tests.
- Thresholds such as CPU 80% or DB connection 80% are general monitoring defaults, not company or customer operating standards.
- This repository does not contain internal structure, data schema, operational secrets, or production data from any commercial APM product.

## Problems This Project Explores

| # | Problem | Approach |
| --- | --- | --- |
| 1 | Repeated manual analysis when abnormal APM metrics appear | Threshold detection plus LangChain/Ollama analysis |
| 2 | Verifying the quality of the AI analysis itself | DeepEval-based Hallucination, Faithfulness, and Relevancy measurement |

The second point is the main focus. The project is less about "using an LLM" and more about adding a QA mechanism around the LLM's output.

## AI Quality Evaluation

DeepEval is used to measure whether the LLM output is grounded in the provided monitoring data.

| Metric | Purpose |
| --- | --- |
| Hallucination | Checks whether the answer invents unsupported facts |
| Answer Relevancy | Checks whether the answer responds to the anomaly context |
| Faithfulness | Checks whether the answer stays faithful to the monitoring data |
| Custom score | Combines expected anomaly type, keyword coverage, and answer completeness |
| Overall quality score | Weighted average of custom and DeepEval metrics |

From a QA perspective, this matters because plausible AI output is not automatically correct. The project makes that gap visible and measurable.

## Limitations

- This is not a production-ready AIOps solution.
- Monitoring data is mock data and the threshold logic is intentionally simple.
- The selected LLM model is for learning and experimentation, not a validated production recommendation.
- DeepEval scores are not absolute truth. Samples should still be reviewed by a human.
- Mattermost alerts are one-way; user feedback is not yet fed back into model selection or evaluation.
